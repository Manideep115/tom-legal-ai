# 🏛️ AI Lawyer Pro – Final Project Report

## (Advanced Legal RAG System with Hybrid Retrieval)

---

# 📌 1. Introduction

The rapid advancement of Large Language Models (LLMs) has enabled intelligent systems capable of understanding and generating human-like text. However, in domains such as **law**, accuracy, traceability, and reliability are critical. Standard LLM-based systems often suffer from **hallucinations**, lack of **source attribution**, and inability to handle **structured legal reasoning**.

This project, **AI Lawyer Pro**, was developed to address these limitations by building a **Retrieval-Augmented Generation (RAG)** system specifically tailored for legal document understanding. The system evolved through multiple iterations, incorporating improvements such as **semantic retrieval, reranking, hallucination control, and Graph-based reasoning**.

---

# 🎯 2. Problem Statement

The primary objective of this project is:

> To design and implement an AI system capable of analyzing legal documents and answering queries with **high accuracy, contextual relevance, and verifiable sources**.

### Key Challenges Identified:

* Legal datasets are **complex and unstructured**
* Existing datasets lack **critical metadata**
* LLMs tend to:

  * Generate **incorrect legal references**
  * Provide **non-verifiable answers**
* Difficulty in handling:

  * Multi-step legal reasoning
  * Large document context

---

# 🧱 3. Initial Approach and Limitations

### 3.1 Dataset Selection (Initial Attempt)

The project initially used **CSV-based datasets** containing IPC and CrPC sections.

### Issues Faced:

* Missing attributes such as:

  * Bailable / Non-bailable classification
  * Procedural details
* Lack of completeness and reliability

### Decision Taken:

The dataset source was shifted to **Official Bare Act PDFs**, ensuring:

* Authenticity
* Completeness
* Legal correctness

---

# ⚙️ 4. Data Engineering and Preprocessing

### 4.1 Text Extraction

* Tool used: `PyPDF2`
* Extracted raw text from legal PDFs

### 4.2 Section Segmentation

Since legal documents are not cleanly structured:

* Implemented **Regex-based parsing**
* Identified section boundaries precisely

### 4.3 Data Structuring

Each section was transformed into structured format:

* Section number
* Content
* Legal metadata

---

# ✂️ 5. Chunking Strategy

### Approach:

* Chunk size: 1000 characters
* Overlap: 200 characters

### Purpose:

* Preserve semantic continuity
* Avoid splitting legal definitions

### Challenge:

* Ensuring chunks do not break logical meaning

### Outcome:

* Improved context retention for retrieval

---

# 🧠 6. Semantic Representation (Embeddings)

### Model Used:

* `SentenceTransformers (all-MiniLM-L6-v2)`

### Purpose:

* Convert legal text into vector representations
* Enable semantic similarity search

---

# 🗄️ 7. Vector Database Implementation

### Database:

* ChromaDB

### Stored Information:

* Text chunks
* Metadata:

  * law_type
  * section
  * classification

### Benefit:

* Fast and efficient semantic retrieval

---

# 🔍 8. Basic RAG Pipeline

### Workflow:

1. User query → embedding
2. Search in vector DB
3. Retrieve top relevant chunks
4. Pass to LLM for answer generation

### Limitation Observed:

* Retrieved results were not always optimally ranked
* Some irrelevant chunks appeared in top results

---

# 🎯 9. Reranking Mechanism (Improvement Phase)

### Solution Implemented:

* Cross-Encoder: `ms-marco-MiniLM-L-6-v2`

### Function:

* Re-evaluates retrieved chunks
* Ranks them based on deeper relevance

### Result:

* Improved precision
* Better answer quality

---

# 🤖 10. LLM Integration

### Models Used:

* Groq API (Llama-3.1-8b-instant)
* Ollama (local fallback)

### Role:

* Generate final responses based on retrieved context

---

# 🚨 11. Hallucination Problem and Solution

### Problem:

LLMs occasionally:

* Generate incorrect section numbers
* Provide fabricated legal references

### Solution:

* Implemented **Regex-based validation system**
* Cross-checked LLM output with database metadata

### Outcome:

* Ensured factual correctness
* Reduced hallucination significantly
