## 📌 Phase 1: Data Engineering & Preprocessing (The ETL Pipeline)
*The foundation of the project was ensuring the AI had access to highly accurate, uncorrupted legal text.*
* **Data Sourcing:** We initially evaluated CSV datasets for IPC and CrPC but discovered they lacked crucial procedural metadata (e.g., Bailable vs. Non-Bailable status). We pivoted to using the **Official Bare Act PDF** for maximum accuracy.
* **Extraction & Transformation:** Used `PyPDF2` to extract raw text. Wrote custom Python logic (Regex) to intelligently split the massive PDF precisely where each legal "Section" began.
* **Data Enrichment (Metadata Injection):** Mapped critical procedural rules (Cognizable/Bailable status, Punishment duration) into the data. 
* **Chunking Strategy:** Split the legal text into chunks of 1000 characters with a 200-character overlap. This ensured no legal definition was cut in half, preserving context for the AI.

## 📌 Phase 2: Building the Vector Database (The Legal Brain)
*Instead of keyword matching, we built a semantic search engine that understands the "meaning" of legal queries.*
* **Embedding Model:** Utilized `SentenceTransformers` (`all-MiniLM-L6-v2`) to convert human text and legal jargon into dense mathematical vectors.
* **Vector Storage:** Deployed **ChromaDB** as the persistent local vector database.
* **Metadata Tagging:** Every chunk stored in the database was tagged with metadata (`law_type`, `section`, `classification`). This allows the AI to filter answers based on legal categories instantly.

## 📌 Phase 3: Advanced RAG Architecture (Backend API)
*We built a robust backend using **FastAPI** to handle the heavy AI lifting and logic.*
* **Initial Retrieval:** When a user asks a question, ChromaDB fetches the top most relevant legal chunks.
* **Reranking (Sniper Logic):** Implemented a `Cross-Encoder` (`ms-marco-MiniLM-L-6-v2`) to re-evaluate and sort the retrieved documents, pushing the most accurate legal sections to the absolute top.
* **LLM Synthesis:** Integrated the **Groq API (Llama-3.1-8b-instant)** for lightning-fast, highly accurate generation, alongside **Ollama** for local, offline model support.
* **Hallucination Prevention (Citation Filter):** Built a strict Regex verification system. The AI is forced to cite its sources, and the system cross-checks the LLM's output against the ChromaDB metadata. If the AI hallucinates a section number, it is flagged.

## 📌 Phase 4: User Interface Development (Frontend)
*We built an interactive, accessible web app for end-users using **Streamlit**.*
* **Authentication System:** Built a secure login/signup gatekeeper using session states and JSON storage to keep user chats private.
* **Dynamic Chat Interface:** Created a ChatGPT-style conversational UI that displays the AI's answer, followed dynamically by a "Sources" tab showing exactly which IPC/CrPC sections were used.
* **Microservices Architecture:** Designed the frontend to be completely separate from the backend, communicating purely via RESTful API calls (`requests.post(BASE_URL)`).

## 📌 Phase 5: Deployment & Environment Isolation
*Ensured the project was stable, reproducible, and accessible over the internet.*
* **Virtual Environment (`venv`):** Isolated all project dependencies (FastAPI, Torch, ChromaDB) into a secure sandbox to prevent version conflicts.
* **Multi-Threaded Execution:** Ran the project using asynchronous server execution:
    * `uvicorn` running the Backend on port 8000.
    * `streamlit` running the Frontend on port 8501.
* **Live Web Tunneling:** Implemented **Ngrok** to create a secure tunnel, exposing the local Streamlit application to the public internet for live demonstration and testing.

## 🚀 Future Upgrades (To impress your guide)
* **New Criminal Laws Update:** Transitioning the database from IPC/CrPC to the newly enacted **BNS (Bharatiya Nyaya Sanhita)** and **BNSS**.
* **GraphRAG Integration:** Upgrading from standard Vector search to a Knowledge Graph (Neo4j) to handle highly complex, multi-hop legal reasoning.
* **Hybrid Search:** Combining semantic search with BM25 keyword search to accurately catch exact Latin legal maxims.

### **Project Development Lifecycle: Legal AI Dashboard ("Tom")**

#### **Phase 1: Architecture Design & Tech Stack Selection**
* **Objective:** To build a high-precision, hallucination-free AI assistant capable of analyzing First Information Reports (FIRs) and legal documents using the newly enacted Indian Criminal Laws (BNS, BNSS, BSA).
* **Core Technologies Selected:** * **Frontend:** Streamlit (for rapid UI deployment and interactive dashboards).
    * **AI/LLM:** OpenRouter API (acting as the reasoning engine).
    * **Databases:** ChromaDB (Vector Search) and NetworkX (Graph Search).
    * **Data Processing:** `pdfplumber` (text extraction), `pytesseract` & `poppler` (OCR for scanned PDFs).
    * **Visualization:** Plotly (`plotly.express` & `plotly.graph_objects`).

#### **Phase 2: Data Engineering & Ingestion Pipeline**
* **Master Legal Knowledge Base:** Compiled the BNS, BNSS, and BSA into a highly structured JSON format (`master_legal_v1.json`) to serve as the ground truth.
* **Robust Document OCR:** Engineered a dual-fallback document reader. The system first attempts standard PDF text extraction; if it detects a scanned image (under 50 characters of text), it automatically boots up Tesseract OCR to read the images.

#### **Phase 3: Building the "Hybrid" RAG Engine (The Brain)**
* **Vector Search (Semantic):** Embedded the Indian legal acts into a local ChromaDB instance to allow the AI to search for laws based on the "meaning" of the words in the uploaded FIR.
* **GraphRAG Implementation (Relational):** Built a Directed Multigraph (`legal_graph.json`) with over 1,140 nodes and 1,860 edges. 
    * *Why this matters:* While Vector Search finds individual laws, the Graph structure allows the AI to understand hierarchical relationships (`PART_OF_CHAPTER`) and multi-hop legal dependencies (`REFERENCES`), giving it superior contextual awareness.

#### **Phase 4: Advanced Prompt Engineering & AI Constraints**
* **Zero-Creativity Enforcement:** Set the LLM `temperature` to `0.0` to force deterministic, factual outputs, completely eliminating the LLM's tendency to invent fake laws or "social" definitions.
* **Relevancy Filtering:** Programmed the system prompt to cross-reference retrieved laws against the actual document facts, instructing the AI to actively ignore irrelevant search results (Context Contamination).
* **Strict Schema Extraction:** Utilized "Few-Shot Prompting" by injecting a hard-coded JSON blueprint inside `[[DATA]]` tags. This forced the LLM to output highly predictable, formatted data (Timeline, Risk Score, Section Impact) rather than conversational text.

#### **Phase 5: Interactive Dashboard & UI Development**
* **Custom CSS Styling:** Overcame standard Streamlit limitations by writing custom, responsive CSS (Flexbox) to create a sticky, right-aligned chat interface that doesn't overlap with the main data dashboard.
* **Dynamic Visualizations:** Passed the LLM-generated JSON directly into Plotly to create real-time analytics:
    * **Scatter Plot:** For mapping the chronological timeline of the legal case.
    * **Gauge Chart:** For calculating and displaying the Case Severity/Risk Score.
    * **Horizontal Bar Chart:** For visualizing the relevance and impact of specific legal sections.

#### **Phase 6: Error Handling & System Hardening**
* **Schema Drift Correction:** Implemented Python-side safety checks (e.g., auto-renaming incorrect LLM keys like `"timestamp"` to `"date"`) to prevent Pandas and Plotly from throwing `ValueErrors`.
* **Crash Prevention:** Added logic to gracefully handle empty data frames and missing API keys via `.env` path routing, ensuring the application remains stable even if the LLM fails to generate perfect data.
